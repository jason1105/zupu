# 产品需求文档（PRD）

**项目名称**：族谱（Zupu）  
**版本**：1.0.0  
**日期**：2026-04-18  
**作者**：Claude  

---

## 1. 产品背景

家族历史是民族文化的重要组成部分。传统纸质族谱难以共享、难以更新、容易损毁。本产品旨在将族谱数字化，让家族成员能够在线协作管理、查阅和传承家族历史。

---

## 2. 目标用户

| 角色 | 描述 |
|------|------|
| 家族创建者 | 发起建立家族族谱，担任首个管理员 |
| 家族管理员 | 负责录入和维护成员信息及关系 |
| 家族成员 | 注册后可查阅家族族谱 |

---

## 3. 用户故事

### 3.1 认证

- **US-01**：作为访客，我希望能注册账号，以便管理我的家族族谱。
- **US-02**：作为用户，我希望能用邮箱和密码登录，以便访问我的族谱数据。
- **US-03**：作为用户，我希望能安全登出，以便保护我的账号隐私。

### 3.2 家族管理

- **US-04**：作为已登录用户，我希望能创建新家族，并成为该家族的管理员。
- **US-05**：作为家族管理员，我希望能修改家族名称和描述。
- **US-06**：作为家族创建者，我希望能指定其他用户为该家族的管理员，以便分担管理工作。
- **US-07**：作为家族管理员，我希望能删除空家族（无成员时）。
- **US-08**：作为已登录用户，我希望能查看自己所属和管理的所有家族列表。

### 3.3 成员管理

- **US-09**：作为家族管理员，我希望能添加家族成员，包含姓名、性别、生日、职业、籍贯、照片等信息。
- **US-10**：作为家族管理员，我希望能标记成员为"在世"或"已故"，已故成员可录入忌日。
- **US-11**：作为家族管理员，我希望能编辑已有成员信息。
- **US-12**：作为家族管理员，我希望能删除成员（同时删除其所有关系记录）。
- **US-13**：作为任意已登录用户，我希望能查看族谱成员的详细信息。

### 3.4 关系管理

- **US-14**：作为家族管理员，我希望能建立"父/母 → 子/女"关系，以表达亲子关系。
- **US-15**：作为家族管理员，我希望能建立"夫妻"关系，以表达配偶关系。
- **US-16**：作为家族管理员，我希望能删除错误的关系记录。

### 3.5 族谱可视化

- **US-17**：作为用户，我希望能以树状图查看家族成员的代际关系，以便直观了解族谱结构。
- **US-18**：作为用户，我希望能在树状图上缩放和平移，以便查看大型族谱。
- **US-19**：作为用户，我希望能点击树状图中的成员节点，查看其详细信息。
- **US-20**：作为用户，我希望能以列表形式查看所有成员，并能按姓名、性别、在世状态筛选。

### 3.6 数据导入导出

- **US-21**：作为家族管理员，我希望能将族谱导出为 JSON 文件（包含成员和关系的完整数据）。
- **US-22**：作为家族管理员，我希望能将成员列表导出为 CSV 文件。
- **US-23**：作为家族管理员，我希望能从 JSON 文件导入族谱数据，以便迁移或恢复数据。

## 4. 功能需求

### 4.1 认证模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-01 | 邮箱 + 密码注册，密码至少 8 位 | P0 |
| F-02 | 邮箱 + 密码登录 | P0 |
| F-03 | 安全登出，清除 Session | P0 |
| F-04 | 密码加密存储（bcrypt） | P0 |

### 4.2 家族模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-05 | 创建家族（名称必填，描述选填） | P0 |
| F-06 | 编辑家族基本信息 | P0 |
| F-07 | 删除家族（无成员时方可删除） | P1 |
| F-08 | 按邮箱查找用户并指定为管理员 | P1 |
| F-09 | 移除管理员（创建者不可被移除） | P1 |
| F-10 | 用户家族列表（我创建的 + 我管理的 + 我加入的） | P0 |

### 4.3 成员模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-11 | 创建成员（姓名、性别必填） | P0 |
| F-12 | 成员字段：姓名、性别、生日、忌日、在世/已故、职业、籍贯、照片URL、简介 | P0 |
| F-13 | 编辑成员信息 | P0 |
| F-14 | 删除成员（级联删除其关系） | P0 |
| F-15 | 已故成员在UI上有视觉区分（灰色/标记） | P1 |

### 4.4 关系模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-16 | 创建 PARENT_CHILD 关系（指定父/母和子/女） | P0 |
| F-17 | 创建 SPOUSE 关系（双向，不重复） | P0 |
| F-18 | 删除关系 | P0 |
| F-19 | 防止循环关系（A 是 B 的父亲，B 不能是 A 的父亲） | P1 |
| F-20 | 防止重复关系 | P0 |

### 4.5 可视化模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-21 | SVG 树状图，以根节点为起点向下展开 | P0 |
| F-22 | 节点显示：姓名、性别图标、在世状态 | P0 |
| F-23 | 配偶在节点旁并排显示，连线为虚线 | P1 |
| F-24 | 支持鼠标滚轮缩放、拖拽平移 | P0 |
| F-25 | 点击节点弹出成员详情卡片 | P0 |
| F-26 | 列表视图，支持按字段筛选 | P0 |

### 4.6 导入导出模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-27 | 导出 JSON（完整族谱：成员 + 关系） | P0 |
| F-28 | 导出 CSV（成员基本信息） | P0 |
| F-29 | 导入 JSON（创建新成员和关系，不覆盖已有数据） | P1 |

## 5. 非功能需求

| 类别 | 需求 |
|------|------|
| 性能 | 页面首屏加载 < 3秒；500名成员的树状图渲染 < 1秒 |
| 安全 | 密码 bcrypt 加密；API 接口鉴权；防止 SQL 注入（Prisma ORM 参数化查询） |
| 可用性 | 响应式设计，支持桌面和平板；操作有加载和错误反馈 |
| 可维护性 | TypeScript 全栈；Prisma 管理数据库迁移；单元测试覆盖核心逻辑 |
| 数据安全 | 不同家族数据隔离；只有管理员可修改数据 |

## 6. 界面原型描述

### 登录/注册页
- 居中表单，含 Logo
- 登录：邮箱、密码输入框，登录按钮，跳转注册链接
- 注册：姓名、邮箱、密码、确认密码，注册按钮

### 仪表盘
- 顶部导航栏（Logo、用户名、登出）
- 家族卡片网格（每卡片含家族名、成员数、我的角色）
- "创建新家族"按钮

### 家族详情页
- 顶部：家族名、描述、"树状视图 / 列表视图"切换 Tab
- 树状视图：全屏 SVG 画布，支持缩放平移
- 列表视图：成员表格，含操作列（编辑/删除）
- 右侧浮动按钮（管理员可见）：添加成员、添加关系、导入导出

### 成员表单
- 模态框或独立页面
- 字段分组：基本信息（必填）、详细信息（选填）、照片

---

## 7. 验收标准

1. 用户可以完整走通"注册 → 登录 → 创建家族 → 添加成员 → 建立关系 → 查看树状图"的主流程。
2. 导出的 JSON 文件可以被成功导入到新家族中，数据无损。
3. 所有 P0 功能需求均已实现并通过测试。
4. 单元测试覆盖率 ≥ 70%（核心工具函数和 API 路由）。
5. 集成测试覆盖主要用户流程。
