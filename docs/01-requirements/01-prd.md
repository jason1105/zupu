# 产品需求文档（PRD）

**项目**：族谱（Zupu）  
**版本**：1.1.0  
**更新日期**：2026-04-26  

---

## 1. 产品背景

家族历史是民族文化的重要组成部分。传统纸质族谱难以共享、难以更新、容易损毁。本产品旨在将族谱数字化，让家族成员能够在线协作管理、查阅和传承家族历史。

---

## 2. 目标用户

| 角色 | 描述 |
|------|------|
| 家族创建者 | 发起建立家族族谱，担任首个管理员，拥有最高权限 |
| 家族管理员 | 负责录入和维护成员信息及关系，可审批加入申请 |
| 普通用户 | 注册后可搜索家族并申请加入，加入后可查阅族谱 |

---

## 3. 用户故事

### 3.1 认证

- **US-01**：作为访客，我希望能注册账号，以便管理我的家族族谱。
- **US-02**：作为用户，我希望能用邮箱和密码登录，以便访问我的族谱数据。
- **US-03**：作为用户，我希望能安全登出，以便保护我的账号隐私。

### 3.2 家族管理

- **US-04**：作为已登录用户，我希望能创建新家族并成为管理员。
- **US-05**：作为家族管理员，我希望能修改家族名称和简介。
- **US-06**：作为家族创建者，我希望能指定其他用户为管理员，以便分担管理工作。
- **US-07**：作为已登录用户，我希望能查看自己管理的所有家族列表。
- **US-08**：作为已登录用户，我希望能搜索平台上所有家族，以便发现并申请加入感兴趣的家族。

### 3.3 加入申请

- **US-09**：作为普通用户，我希望能向家族提交加入申请（含真实姓名和至少 20 字理由）。
- **US-10**：作为家族管理员，我希望能查看、批准或拒绝加入申请。
- **US-11**：作为申请人，我希望申请被拒绝后能重新提交申请。

### 3.4 成员管理

- **US-12**：作为家族管理员，我希望能添加家族成员，包含姓名、性别、生日、职业、籍贯、照片等信息。
- **US-13**：作为家族管理员，我希望能标记成员为"在世"或"已故"，已故成员可录入忌日。
- **US-14**：作为家族管理员，我希望能编辑和删除成员信息。

### 3.5 关系管理

- **US-15**：作为家族管理员，我希望能建立亲子关系和夫妻关系。
- **US-16**：作为家族管理员，我希望能删除错误的关系记录。

### 3.6 族谱可视化

- **US-17**：作为用户，我希望以树状图查看代际关系，直观了解族谱结构。
- **US-18**：作为用户，我希望能在树状图上缩放和平移。
- **US-19**：作为用户，我希望能点击节点查看成员详细信息。
- **US-20**：作为用户，我希望能以列表形式查看成员，并按姓名和在世状态筛选。

### 3.7 数据导入导出

- **US-21**：作为家族管理员，我希望能导出 JSON（完整数据）和 CSV（成员列表）。
- **US-22**：作为家族管理员，我希望能从 JSON 文件导入族谱数据。

## 4. 功能需求

### 4.1 认证模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-01 | 邮箱 + 密码注册，密码至少 8 位，bcrypt 加密存储 | P0 |
| F-02 | 邮箱 + 密码登录，错误信息统一（防止账号枚举） | P0 |
| F-03 | 安全登出，清除 JWT Session | P0 |

### 4.2 家族模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-04 | 创建家族（名称必填，简介必填） | P0 |
| F-05 | 编辑家族名称和简介 | P0 |
| F-06 | 按关键字搜索所有家族（含成员数、用户数） | P0 |
| F-07 | 按邮箱添加管理员（创建者专属权限） | P1 |
| F-08 | 移除管理员（创建者不可被移除） | P1 |

### 4.3 加入申请模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-09 | 提交加入申请（真实姓名必填，理由 ≥ 20 字） | P0 |
| F-10 | 已是管理员时禁止申请（返回 409） | P0 |
| F-11 | REJECTED 状态可重新申请（更新现有记录） | P1 |
| F-12 | 管理员查看申请列表（分 PENDING / 已处理） | P0 |
| F-13 | 管理员批准申请（自动添加为管理员） | P0 |
| F-14 | 管理员拒绝申请 | P0 |

### 4.4 成员模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-15 | 创建成员（姓名、性别必填；生日、忌日、职业、籍贯、照片、简介选填） | P0 |
| F-16 | 编辑成员信息 | P0 |
| F-17 | 删除成员（级联删除其关系） | P0 |
| F-18 | 已故成员在 UI 上有视觉区分 | P1 |

### 4.5 关系模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-19 | 创建 PARENT_CHILD 关系（fromMember 为父/母，toMember 为子/女） | P0 |
| F-20 | 创建 SPOUSE 关系（双向，唯一约束防重复） | P0 |
| F-21 | 删除关系 | P0 |

### 4.6 可视化模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-22 | D3.js SVG 树状图，根节点在顶，向下展开 | P0 |
| F-23 | 节点显示姓名、性别图标、在世状态 | P0 |
| F-24 | 配偶并排显示，虚线连接 | P1 |
| F-25 | 鼠标滚轮缩放、拖拽平移 | P0 |
| F-26 | 点击节点显示成员详情面板 | P0 |
| F-27 | 列表视图，按姓名和在世状态筛选 | P0 |

### 4.7 导入导出模块

| 编号 | 需求 | 优先级 |
|------|------|--------|
| F-28 | 导出 JSON（成员 + 关系完整数据，带 UTF-8 BOM） | P0 |
| F-29 | 导出 CSV（成员基本信息，带 UTF-8 BOM，兼容 Excel） | P0 |
| F-30 | 导入 JSON（Prisma 事务，失败全部回滚） | P1 |

---

## 5. 非功能需求

| 类别 | 需求 |
|------|------|
| 性能 | 500 名成员的树状图渲染 < 1 秒 |
| 安全 | bcrypt 加密密码；API 全部鉴权；Prisma 参数化查询防 SQL 注入 |
| 可用性 | 响应式设计，适配桌面和平板；操作有加载和错误反馈 |
| 可维护性 | TypeScript 全栈；Prisma 管理迁移；Vitest 单元 + 集成测试 |
| 数据隔离 | 不同家族数据严格隔离；只有管理员可写数据 |

---

## 6. 验收标准

1. 完整主流程：注册 → 登录 → 创建家族 → 添加成员 → 建立关系 → 查看树状图。
2. 加入申请流程：用户 B 搜索家族 → 提交申请 → 用户 A 批准 → 用户 B 成为管理员。
3. 导出 JSON 可成功导入到新家族，数据无损。
4. 所有 P0 功能需求均已实现并通过测试。
5. `npm test` 全部通过（当前 36 个测试）。
